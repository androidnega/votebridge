1. EXECUTIVE SUMMARY
================================================================================

VoteBridge is a multi-phase campus e-voting platform comprising 17 Django backend
apps, 39 database models, ~175 REST API endpoints, a Vue 3 SPA (primary UI),
and a parallel legacy Alpine.js + Django template frontend. The system has
reached enterprise breadth: authentication with OTP/MFA, elections lifecycle,
voting, results certification, strongroom integrity, fraud monitoring, operations
center, communications, USSD, analytics, biometrics, trusted devices, and a
Super Admin System Control Center.

MATURITY ASSESSMENT: Late-stage / feature-rich beta. Core voting paths are
implemented end-to-end. Multiple enterprise modules coexist with legacy pages,
partial Vue coverage, sidebar/route drift, and overlapping dashboards.

KEY COUNTS:
  Backend Django apps ............................ 17
  Backend models ................................. 39 (13 apps with models)
  Backend service modules ........................ 55 (+1 core realtime)
  Backend repository modules ..................... 21
  Backend permission classes ..................... 42 (incl. aliases)
  REST API endpoint patterns (app urls) ............ ~175
  Legacy Django HTML templates ................... 35
  Legacy dashboard URL routes .................... 33
  Vue router records ............................. 79 (76 named + 3 redirects)
  Vue view files ................................. 80
  Vue Pinia stores ............................... 15
  Vue API client modules ......................... 18
  Vue reusable components ........................ 72
  Documented phase reports ....................... 7 formal + 1 overview + 1 ERD
  Backend TODO/FIXME markers ..................... 0
  Frontend TODO/FIXME markers .................... 0

ARCHITECTURE COMPLIANCE (summary): Predominantly compliant (Views -> Services ->
Repositories -> Models). Documented violations: 2 USSD API views with direct ORM
access. Several apps have no models (analytics, dashboard, operations, realtime)
by design (aggregation/monitoring layers).

DUAL-FRONTEND STATE: Vue SPA (frontend/index.html -> src/vue/) is the modern UI.
Legacy stack (frontend/src/main.js, Alpine.js, backend/templates/) remains active
for election CRUD, security pages, fraud pages, and voting flows accessible via
/dashboard/* routes.

2. USER ROLE INVENTORY (PART 2)
================================================================================

DATABASE ROLES (accounts.Role):
  - super_admin   (Super Admin)
  - admin         (Admin)
  - student       (Student)
  - candidate     (Candidate)

NOTE: User query listed Super Admin, Admin, Student only. Candidate exists as a
distinct role in schema. "Election Officer" is documented as using Admin role;
no separate election_officer role in database.

--------------------------------------------------------------------------------
2.1 SUPER ADMIN
--------------------------------------------------------------------------------

RESPONSIBILITIES (as implemented):
  - Full platform configuration (System Control Center)
  - Results certification, publication, archiving
  - Voting channel management (Super Admin only write)
  - Biometric enrollment
  - Feature flags, backups, maintenance, institution/branding
  - All Admin capabilities

ACCESSIBLE VUE PAGES (route meta roles: super_admin OR unrestricted within auth):
  Dashboard: /, /dashboard/super-admin
  Elections: /elections, /elections/:uuid (manage; no vote routes)
  Results: /results, /results/certification, /results/publication, /results/archive,
           /results/:electionUuid
  Strongroom: /strongroom, /strongroom/:electionUuid
  Operations: all /operations/* (10 views; sidebar shows subset)
  Platform logs: /platform/logs
  Analytics: all /analytics/* (15 views)
  Communications: all /communications/*
  USSD: /ussd, /ussd/sessions
  Security: /security, /security/trusted-devices
  Fraud: /fraud (no route-level role gate)
  Notifications: /notifications
  System Control: all /system-control/* (17 views) + /biometrics/enroll,
                  /biometrics/history
  Profile: /profile

SIDEBAR (superAdminNav): Command center, Elections, Results, Strongroom,
  Operations (4 children), Platform logs, Analytics, Security center,
  Communications (3 children), USSD, System control (7 children), Notifications

KEY APIs:
  /api/v1/system/* (exclusive)
  /api/v1/results/*/certify, publish, archive
  /api/v1/elections/channels/ (write)
  /api/v1/biometrics/enrollment/
  /api/v1/trusted-devices/<uuid>/ (revoke any via CanRevokeAnyDevice)
  All admin-accessible endpoints below

DUPLICATED PRIVILEGES vs ADMIN:
  - Election management, candidates, positions, eligibility (identical)
  - Operations center, analytics, communications, USSD monitoring (identical)
  - Strongroom view/manage (identical)
  - Fraud and security monitoring (identical)
  - Trusted devices management page (both admin + super_admin)

PRIVILEGES UNIQUE TO SUPER ADMIN:
  - System Control Center (CanAccessSystemControlCenter)
  - Results certification/publication/archive queues
  - Voting channel create/update/delete
  - Biometric enrollment
  - Communication provider management (CanManageCommunications)
  - Trusted device revoke-any, biometric/trusted device settings

MISSING RESTRICTIONS (observations):
  - /fraud has no meta.roles; students could navigate if URL known (guard only
    checks requiresAuth)
  - /security has no meta.roles; same pattern
  - /notifications has no meta.roles; students have sidebar link

  - Strongroom viewing and integrity actions

ACCESSIBLE VUE PAGES:

  - No /system-control/*
  - No /biometrics/enroll
  - No /results/certification, /results/publication, /results/archive
  - Uses /dashboard/admin instead of super-admin dashboard

SIDEBAR (adminNav): Command center, Elections, Results, Strongroom, Operations,
  Platform logs, Analytics, Security center, Communications, USSD, Notifications
  (no System control group)

KEY APIs:
  /api/v1/elections/* (except channel write if super_admin only)
  /api/v1/candidates/*
  /api/v1/operations/*
  /api/v1/analytics/* (not personal-only endpoint semantics)
  /api/v1/fraud/*
  /api/v1/security/monitoring/*
  /api/v1/strongroom/*
  /api/v1/notifications/dashboard, deliveries, queue (not CanManageCommunications)
  /api/v1/ussd/dashboard, sessions, logs
  /api/v1/dashboard/admin/

DUPLICATED PRIVILEGES vs SUPER ADMIN:
  - Near-complete overlap on operational/monitoring features
  - Both can open/close elections, manage ballots setup, view analytics

UNNECESSARY PRIVILEGES (observation, not recommendation):
  - Admin has access to 10 operations sub-routes while sidebar exposes 4
  - Admin can access full analytics suite (15 pages) while analyticsNav shows 5

15. OBSERVATIONS (PART 14 — NO RECOMMENDATIONS)
================================================================================

1. The platform implements a full election lifecycle from draft through archive
   with integrity controls (SVT, strongroom, readiness gate, result certification).

2. Super Admin and Admin roles overlap substantially on operational features;
   differentiation is mainly System Control, results certification, and channel mgmt.

3. Navigation has been streamlined in sidebar (Phase UI-X / super admin consolidation)
   but router still registers many pages not linked from sidebar (analytics subpages,
   operations subpages, system-control subpages).

4. Two parallel frontends coexist: Vue SPA for dashboards/monitoring; Django
   templates for election setup and some security/fraud pages.

5. Platform logs consolidation (PlatformLogsView) replaced three log views in
   router but orphan Vue files remain.

6. Feature flags include 3 "future_*" reserved flags still togglable in UI.

7. Zero TODO/FIXME markers in codebase despite known gaps (user mgmt UI, personal
   analytics page, Vue election create).

8. Analytics module is read-only with no models — consistent architecture for BI layer.

9. Trusted devices and biometrics add significant identity assurance surface area
   with 12+ trusted device services and 7 biometrics services.

10. system_service.py concentration (10 classes) is the largest single-file service
    bundle in the project.

11. Student role has minimal sidebar (3 items) aligned with voting use case.

12. Candidate role exists in DB; UI treats candidates like students for voting.

13. Public verification (/verify) and maintenance (/maintenance) are the only
    significant public-facing Vue routes outside auth.

14. WebSocket realtime is used across dashboard, operations, security, fraud,
    analytics, strongroom, and election feeds.

15. Configuration is extensive: 88+ individual setting keys across 11 categories
    plus 11 feature flags and institution profile fields.
